# ADR-0003: Backend technology stack and repo boundary

Status: Accepted
Date: 2026-07-09

## Context

`doc.md` fixes most of the backend stack already: NestJS, PostgreSQL,
Prisma, Redis, and a decision to **not** use a monorepo — front and back
live in separate GitHub repositories. This ADR records the reasoning so
future contributors don't reintroduce a monorepo or a shared-package
dependency between the two repos by default.

## Decision

- **Framework:** NestJS (Node.js/TypeScript). Modular structure (domain
  modules: users, boards, tasks, finance, loans), native DI, first-class
  OpenAPI/Swagger generation via `@nestjs/swagger`.
- **Database:** PostgreSQL, hosted on **Neon or Supabase** (both have a
  usable free tier; pick whichever also ends up hosting Storage/Realtime,
  see [ADR-0006](0006-realtime-communication.md), to reduce the number of
  vendors).
- **ORM:** Prisma. Schema-first models, migrations, and generated types
  used directly by NestJS services.
- **Cache / queues:** Redis via **Upstash** (free tier, serverless-friendly,
  no server to manage). Used for caching, and as the backing store for a
  job queue (BullMQ) for document ingestion — see
  [ADR-0007](0007-financial-document-ingestion.md).
- **Auth:** Custom JWT issuance via **Passport.js** strategies inside
  NestJS — not a third-party auth service. Detailed in
  [ADR-0004](0004-authentication-and-authorization.md).
- **Repo boundary — no monorepo:** `auctoritas-back` and `auctoritas-front`
  are independent repositories. The **only** contract between them is a
  versioned **OpenAPI/Swagger spec**, generated by NestJS from decorated
  controllers/DTOs. The frontend consumes it by generating a typed TS
  client with **orval**, checked into `auctoritas-front` (or generated in
  CI) rather than importing backend source directly.

### Why no monorepo

Given the repos already exist as separate GitHub projects with independent
histories (`auctoritas-back`, `auctoritas-front`), and the team also plans
separate mobile and browser-extension repos, a shared-package monorepo
would require merging these histories and adds coordination overhead
(shared build tooling, versioning, CI across all clients) that isn't
justified while the team is small. Contract-based integration via
OpenAPI keeps each repo deployable and versioned independently, at the
cost of an explicit client-generation step whenever the API contract
changes.

## Alternatives considered

- **Monorepo with shared TS types/DTOs:** would remove the OpenAPI/orval
  generation step and guarantee type parity at compile time, but conflicts
  with the already-established separate-repos structure and would require
  restructuring both existing repos. Worth revisiting only if contract
  drift between front and back becomes a recurring source of bugs that
  orval's generated client doesn't catch.
- **GraphQL instead of REST+OpenAPI:** would also give strong contract
  guarantees and flexible querying, but adds a new paradigm the team hasn't
  chosen, and NestJS's OpenAPI generation is more turnkey for a small team.
- **tRPC:** gives end-to-end type safety without codegen, but requires
  shared TypeScript types between client and server — which reintroduces
  the coupling the separate-repos decision was trying to avoid, unless
  published as a versioned package (more moving parts than OpenAPI+orval
  for now).

## Consequences

- Any backend API change that affects the contract must regenerate and
  publish the OpenAPI spec; the frontend must regenerate its client
  (`orval`) before consuming new/changed endpoints — this should be a CI
  step in `auctoritas-front`, not a manual step, to avoid silent drift.
- The backend owns Prisma schema/migrations exclusively; the frontend never
  touches the database directly.
- Choosing Neon vs. Supabase for Postgres, and whether Supabase also covers
  Storage/Realtime, is still open — see
  [ADR-0006](0006-realtime-communication.md) and
  [ADR-0007](0007-financial-document-ingestion.md).
